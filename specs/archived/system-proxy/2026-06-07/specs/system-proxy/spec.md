# System Proxy Integration

## Scope Snapshot

- In scope: автоматическая установка и восстановление системных прокси-настроек ОС для маршрутизации трафика через локальный proxy-listener (SOCKS5/HTTP CONNECT) при запуске/остановке клиента kvn-ws.
- Out of scope: изменение системных DNS-настроек, настройка TUN-интерфейса, управление маршрутизацией ядра.

## Цель

Пользователь клиента kvn-ws хочет, чтобы все приложения, уважающие системные прокси-настройки (HTTP_PROXY, HTTPS_PROXY, NO_PROXY), автоматически маршрутизировали трафик через VPN-туннель без ручной конфигурации каждого приложения. Фича считается успешной, когда при включении туннеля системные прокси устанавливаются на локальный listener, а при выключении — восстанавливаются исходные значения.

## Основной сценарий

1. Пользователь запускает клиент kvn-ws в режиме proxy.
2. Клиент запускает локальный SOCKS5/HTTP CONNECT listener (существующая функциональность).
3. После успешного старта listener клиент устанавливает системные прокси-переменные окружения (HTTP_PROXY, HTTPS_PROXY) и/или OS-level proxy settings, указывающие на `127.0.0.1:<proxy_listen_port>`.
4. Приложения, уважающие системные прокси, начинают маршрутизировать трафик через туннель.
5. При остановке клиента (SIGTERM, SIGINT, краш, отключение) системные прокси восстанавливаются в исходное состояние.
6. NO_PROXY заполняется на основе routing exclude_rules, exclude_ips, exclude_domains.

## User Stories

- P1: Linux-пользователь — установка/восстановление переменных HTTP_PROXY/HTTPS_PROXY/NO_PROXY через env (с возможностью перманентной настройки через systemd environment).
- P2: macOS-пользователь — установка/восстановление системных прокси через `networksetup`.
- P3: Windows-пользователь — установка/восстановление системных прокси через WinHTTP/WinINet API.

## MVP Slice

P1 (Linux env) + базовое восстановление при остановке. Закрывает AC-001, AC-002, AC-003.

## First Deployable Outcome

После первого implementation pass можно запустить клиент на Linux, убедиться что `echo $HTTP_PROXY` показывает `http://127.0.0.1:2310`, а после Ctrl+C — переменная очищена/восстановлена.

## Scope

- Установка и восстановление системных прокси-настроек на уровне ОС.
- Интеграция с lifecycle клиента: установка при старте listener, восстановление при shutdown.
- Формирование NO_PROXY из routing exclude-правил (exclude_ranges, exclude_ips, exclude_domains).
- Поддержка Linux (env vars), macOS (networksetup), Windows (WinHTTP/WinINet).
- Конфигурация: флаг `system_proxy` в client config, опциональные `system_proxy_exclude` override.
- Graceful shutdown: восстановление прокси при SIGTERM/SIGINT.

## Контекст

- Существующий proxy listener запускается только в режиме `mode: proxy`.
- Client config уже содержит `ProxyListen`, `ProxyAuth`, `Routing` с exclude-правилами.
- На Linux системные прокси-переменные влияют только на процессы, запущенные после установки переменных. Дочерние процессы (например, запущенные через systemd) не увидят изменения родительского shell.
- На macOS `networksetup` требует прав root для изменения системных настроек.
- На Windows WinHTTP API не требует прав администратора для текущего пользователя.

## Требования

- RQ-001 Система ДОЛЖНА устанавливать переменные HTTP_PROXY и HTTPS_PROXY (lowercase + uppercase) при запуске proxy listener в режиме `system_proxy: true`.
- RQ-002 Система ДОЛЖНА формировать NO_PROXY на основе routing exclude-правил клиента (exclude_ranges → CIDR в IP-адреса, exclude_domains → домены).
- RQ-003 Система ДОЛЖНА восстанавливать предыдущие значения системных прокси при остановке клиента (SIGTERM, SIGINT).
- RQ-004 На Linux система ДОЛЖНА записывать Environment= строки в файл `/etc/systemd/system/kvn-client.service.d/system-proxy.conf` при наличии прав на запись; в противном случае — логировать warning и продолжить без systemd override.
- RQ-005 На macOS система ДОЛЖНА использовать `networksetup -setwebproxy` и `-setsecurewebproxy` для установки HTTP/HTTPS прокси на активных сетевых интерфейсах.
- RQ-006 На Windows система ДОЛЖНА использовать WinHTTP API (`WinHttpSetDefaultProxyConfiguration`) или реестр для установки системного прокси.

## Вне scope

- Изменение DNS-настроек системы (resolv.conf, systemd-resolved).
- Настройка TUN-интерфейса и маршрутизации ядра (это уже есть в routing package).
- Прокси-аутентификация на уровне системы (только настройка адреса/порта).
- Автоматическое обнаружение PAC-файлов.
- Поддержка FTP_PROXY и других протокольных прокси-переменных (только HTTP/HTTPS).
- Перманентное хранение настроек прокси в конфигурации ОС (настройки действуют только во время работы клиента).

## Критерии приемки

### AC-001 Установка системных прокси на Linux через env vars

- Почему это важно: пользователь Linux получает работающий прокси без ручной настройки каждого приложения.
- **Given** клиент запущен на Linux с `mode: proxy` и `system_proxy: true`
- **When** proxy listener успешно стартует на `127.0.0.1:2310`
- **Then** `HTTP_PROXY`, `http_proxy`, `HTTPS_PROXY`, `https_proxy` установлены в `http://127.0.0.1:2310`
- **Then** `NO_PROXY` и `no_proxy` содержат CIDR-адреса и домены из routing exclude-правил
- Evidence: проверка значений переменных окружения через `/proc/self/environ` процесса клиента.

### AC-002 Восстановление прокси при остановке

- Почему это важно: пользователь не должен оставаться с неработающим прокси после отключения туннеля.
- **Given** клиент работает, системные прокси установлены
- **When** клиент получает SIGTERM или SIGINT (Ctrl+C)
- **Then** `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY` (и lowercase-варианты) возвращены к значениям до запуска клиента
- Evidence: проверка переменных окружения после завершения клиента.

### AC-003 NO_PROXY из routing exclude-правил

- Почему это важно: трафик к исключённым диапазонам/доменам не должен маршрутизироваться через прокси.
- **Given** в конфиге клиента указаны `routing.exclude_ranges: ["10.0.0.0/8"]` и `routing.exclude_domains: ["corp.example.com"]`
- **When** клиент запускается с `system_proxy: true`
- **Then** `NO_PROXY` содержит `10.0.0.0/8, .corp.example.com`
- Evidence: чтение `NO_PROXY` из окружения.

### AC-004 Установка системных прокси на macOS

- Почему это важно: пользователи macOS получают автоматическую настройку прокси.
- **Given** клиент запущен на macOS с `mode: proxy` и `system_proxy: true`
- **When** proxy listener успешно стартует
- **Then** `networksetup -getwebproxy` на активном интерфейсе показывает `Enabled: Yes` и `Server: 127.0.0.1:2310`
- Evidence: вывод `networksetup -getwebproxy <interface>`.

### AC-005 Установка системных прокси на Windows

- Почему это важно: пользователи Windows получают автоматическую настройку прокси.
- **Given** клиент запущен на Windows с `mode: proxy` и `system_proxy: true`
- **When** proxy listener успешно стартует
- **Then** WinHTTP/WinINet прокси-конфигурация указывает на `127.0.0.1:2310`
- Evidence: проверка через `netsh winhttp show proxy`.

### AC-006 Graceful shutdown — восстановление при краше

- Почему это важно: при неожиданном завершении клиента система не должна оставаться с неработающим прокси.
- **Given** клиент работает с установленными системными прокси
- **When** процесс клиента аварийно завершается (SIGKILL, panic)
- **Then** при следующем запуске клиент ДОЛЖЕН проверить, указывает ли системный прокси на его listener, и если да — очистить его (best-effort recovery)
- Evidence: запуск клиента, проверка лога на сообщение о recovery, проверка системных прокси после запуска.

### AC-007 Systemd override при отсутствии прав на запись

- Почему это важно: пользователь без root должен понимать, что systemd override не применён, а не ждать перманентной настройки.
- **Given** клиент запущен на Linux с `mode: proxy` и `system_proxy: true` без права записи в `/etc/systemd/system/kvn-client.service.d/`
- **When** proxy listener успешно стартует
- **Then** в лог выводится warning: `systemd override: permission denied, skipped`
- **Then** `HTTP_PROXY`/`HTTPS_PROXY` установлены (согласно AC-001), systemd override не создан
- Evidence: проверка лога на наличие warning; проверка отсутствия файла `system-proxy.conf`.

## Допущения

- На Linux системные прокси через env vars влияют только на сам процесс клиента и его потомков; для глобального эффекта требуется systemd override. Это приемлемо для MVP.
- Пользователь имеет права на запись в `/etc/systemd/system/kvn-client.service.d/` при использовании systemd-режима.
- На macOS `networksetup` доступен и пользователь имеет права admin/sudo.
- На Windows WinHTTP API доступен без дополнительных прав.
- Системные прокси НЕ настраиваются, если `mode != "proxy"`.

## Критерии успеха

- SC-001 Установка/восстановление прокси занимает < 100ms (не блокирует старт/стоп клиента).
- SC-002 При recovery-запуске клиент не стартует, если не смог очистить "чужой" прокси (защита от конфликтов).

## Краевые случаи

- Системный прокси уже установлен другим процессом до запуска клиента → сохранить исходное значение, восстановить при остановке.
- Несколько запусков клиента → повторная установка не должна удалять оригинальное значение (stack-based restore).
- routing exclude-правила пусты → NO_PROXY пуст или не устанавливается.
- interface list пуст на macOS → пропустить настройку, залогировать warning.
- system_proxy: false (или не указан) → никаких изменений системных настроек.
- Запуск с `mode: tun` → system proxy не применяется (только для proxy mode).

## Открытые вопросы

- Стоит ли реализовывать установку прокси на уровне окружения XDG (Desktop Environments) через gsettings/dconf на Linux? Пока отложено, только env vars + systemd override.
- Как быть с прокси-исключениями для IPv6 адресов? NO_PROXY поддерживает IPv6? На данный момент считаем что да, но требуется проверка на целевых ОС.
