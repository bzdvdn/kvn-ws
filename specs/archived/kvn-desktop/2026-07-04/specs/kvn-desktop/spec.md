# kvn-desktop — Desktop GUI wrapper for kvn-web

## Scope Snapshot

- In scope: кросс-платформенное десктопное приложение с нативным WebView-окном, которое отображает UI kvn-web без браузера.
- Out of scope: изменение архитектуры kvn-web (REST API, SSE, SPA), разделение привилегий UI/client, Android desktop mode.

## Цель

Пользователи Linux, macOS и Windows получают возможность запускать ui kvn-web как нативное десктопное приложение (через иконку в приложениях/пуске), а не только через браузер. При этом браузерный доступ к `http://localhost:2311` сохраняется. На Windows приложение само управляет сервером (старт/стоп), на Linux/macOS — подключается к уже работающему systemd/launchd-сервису.

## Основной сценарий

### Linux/macOS
1. `kvn-web` установлен как systemd-сервис (Linux) или launchd-сервис (macOS) и слушает `127.0.0.1:2311`.
2. Пользователь запускает `kvn-desktop` из меню приложений/пуска.
3. Открывается нативное окно с встроенным WebView, который загружает `http://127.0.0.1:2311`.
4. UI работает идентично браузерной версии (Connect/Disconnect, логи, настройки).
5. При закрытии окна процесс завершается — сервис продолжает работать в фоне.

### Windows
1. Пользователь запускает `kvn-desktop.exe` (или через ярлык в меню Пуск).
2. Если запуск не от администратора — всплывает UAC-запрос на повышение.
3. Приложение поднимает встроенный HTTP-сервер (тот же `webui.Server` из `src/internal/webui/`) на `127.0.0.1:2311`.
4. Открывается нативное WebView2-окно с UI.
5. При закрытии окна: автоматический disconnect, cleanup (сброс system proxy в реестре), останов сервера.
6. Браузерный доступ `http://localhost:2311` также работает (если окно открыто).

## User Stories

Набор user stories не требуется — фича является чистым UX-враппером без новой пользовательской семантики. Подход "просто окно" не создаёт новых сценариев использования.

## MVP Slice

Один бинарник `kvn-desktop`, который:
- На Linux/macOS открывает WebView на `http://localhost:2311`
- На Windows запускает встроенный сервер + WebView2 окно, чистит за собой при закрытии
- Ярлык в приложениях (`.desktop` / `.app` / ярлык Пуск)

## First Deployable Outcome

Бинарник `kvn-desktop` в `bin/`, который можно запустить и увидеть UI kvn-web в нативном окне. На Linux — подключение к systemd-сервису. На Windows — самодостаточный запуск с UAC.

## Scope

- Новый entrypoint: `src/cmd/desktop/main.go` + платформенные файлы `app_linux.go`, `app_darwin.go`, `app_windows.go`
- Использование библиотеки `github.com/webview/webview_go` для WebView
- Флаг `--port` для указания порта kvn-web (по умолчанию 2311)
- Флаг `--server` для ручного указания URL (по умолчанию `http://localhost:2311`)
- Обновление `scripts/build-web.sh` для сборки `kvn-desktop`
- Обновление `scripts/install-web.sh` и `scripts/install-web.ps1` с опциональным `--desktop`
- На Windows: UAC-манифест в .exe или runtime elevation через `shell.Execute`

## Контекст

- `src/internal/webui/` уже содержит полноценный HTTP-сервер с REST API + SSE + embed SPA
- `src/internal/tun/tun.go` — только `//go:build linux`; на Windows/macOS TUN недоступен (stub)
- На Windows приложение работает только в proxy mode (SOCKS5/HTTP CONNECT + system proxy через реестр)
- WebView2 на Windows — системный компонент (поставляется с Windows 11 или устанавливается автоматически); WebKitGTK на Linux — пакет из репозитория
- `kvn-web` при старте сам проверяет `os.Geteuid() != 0` для TUN mode

## Зависимости

- `github.com/webview/webview_go` — кросс-платформенная WebView библиотека
- На Linux: WebKitGTK (webkit2gtk-4.1-dev)
- На Windows: WebView2 Runtime (встроен в Win 11, для Win 10 — автоматический установщик)
- На macOS: WKWebView (встроен в систему)
- Go 1.22+ (build tag `//go:build`)

## Требования

- RQ-001 `kvn-desktop` ДОЛЖЕН открывать системный WebView с UI kvn-web на заданном URL
- RQ-002 На Linux/macOS `kvn-desktop` НЕ ДОЛЖЕН запускать собственный HTTP-сервер — только WebView на существующий `localhost:2311`
- RQ-003 На Windows `kvn-desktop` ДОЛЖЕН при запуске:
  - UAC-повышение до администратора
  - Запуск встроенного `webui.Server` на `127.0.0.1:<port>`
  - Открытие WebView2 окна с UI
- RQ-004 На Windows при закрытии окна ДОЛЖЕН: disconnect, cleanup system proxy, останов сервера
- RQ-005 Бинарник ДОЛЖЕН собираться через `scripts/build-web.sh` (или новый `scripts/build-desktop.sh`)
- RQ-006 Скрипты установки ДОЛЖНЫ поддерживать флаг `--desktop`, добавляющий ярлык в приложения
- RQ-007 Приложение ДОЛЖНО корректно обрабатывать недоступность сервера (показывать экран "сервис не запущен" вместо белого листа)
- RQ-008 Приложение ДОЛЖНО предоставлять кнопку "Restart Service" в окне, которая перезапускает службу kvn-web

## Вне scope

- Модификация React SPA: UI остаётся тем же, никаких desktop-specific изменений
- Системный трей / автозапуск (может быть следующей фичей)
- Native notifictions / toast при коннекте
- Разделение архитектуры UI (user) + client (root) — текущая фича этого не делает
- Android desktop mode
- Установка WebView2 Runtime — считается предусловием (на Win 10+)

## Критерии приемки

### AC-001 Linux: WebView на существующий systemd-сервис

- Почему это важно: основной сценарий на Linux — поверх уже работающего kvn-web в systemd
- **Given** `kvn-web` запущен как systemd-сервис на `127.0.0.1:2311`
- **When** пользователь запускает `kvn-desktop`
- **Then** открывается нативное окно с UI kvn-web (без адресной строки и элементов браузера)
- Evidence: окно отображается, видна панель настроек и логов, кнопка Connect активна

### AC-002 macOS: WebView на существующий launchd-сервис

- Почему это важно: то же поведение, что на Linux, для пользователей macOS
- **Given** `kvn-web` запущен как launchd-сервис на `127.0.0.1:2311`
- **When** пользователь запускает `kvn-desktop.app`
- **Then** открывается нативное окно с UI kvn-web
- Evidence: окно отображается, UI загружается, Connect в proxy mode работает

### AC-003 Windows: самодостаточный запуск с UAC

- Почему это важно: Windows не имеет systemd/launchd — приложение само управляет сервером
- **Given** `kvn-desktop.exe` не запущен, UAC доступен
- **When** пользователь запускает `kvn-desktop.exe` двойным кликом
- **Then** появляется UAC-запрос, после подтверждения открывается окно с UI kvn-web
- Evidence: окно отображается, UI загружается, proxy mode работает

### AC-004 Windows: cleanup при закрытии

- Почему это важно: system proxy (реестр) должен быть сброшен, чтобы не оставался orphaned proxy
- **Given** `kvn-desktop` работает, клиент подключён в proxy mode
- **When** пользователь закрывает окно
- **Then** клиент отключается, system proxy сбрасывается, HTTP-сервер останавливается, процесс завершается
- Evidence: после закрытия в реестре `ProxyEnable` = 0, `ProxyServer` удалён

### AC-005 Кросс-платформа: сборка одного бинарника

- Почему это важно: финальный артефакт — бинарник `bin/kvn-desktop` для каждой платформы
- **Given** код собран через `scripts/build-web.sh` (или `build-desktop.sh`)
- **When** сборка завершена
- **Then** в `bin/` появляется `kvn-desktop` (или `kvn-desktop.exe`)
- Evidence: `ls bin/kvn-desktop*` показывает файл, `file bin/kvn-desktop` показывает корректный ELF/Mach-O/PE

### AC-006 Экран ошибки при недоступном сервере + кнопка запуска

- Почему это важно: пользователь должен понимать, что сервис не запущен, и мочь его запустить из того же окна
- **Given** `kvn-web` не запущен на `localhost:2311`
- **When** `kvn-desktop` запускается и WebView ловит ошибку соединения (connection refused)
- **Then** WebView показывает HTML-страницу с сообщением "Служба kvn-web не запущена" и кнопкой "Запустить службу" (нативную, через bind Go-функции в JS)
- Evidence: вместо белого экрана — информативное HTML-сообщение с кнопкой

### AC-012 Linux/macOS: запуск службы из экрана ошибки

- Почему это важно: пользователь может запустить сервис без терминала
- **Given** на экране ошибки (AC-006), kvn-web не запущен
- **When** пользователь нажимает "Запустить службу"
- **Then** появляется системный диалог пароля (polkit/osascript), после ввода выполняется `systemctl start kvn-web` / `launchctl load`. После успешного запуска WebView перезагружает `localhost:2311`
- Evidence: после ввода пароля сервис запускается, UI загружается

### AC-007 Скрипт установки с флагом --desktop

- Почему это важно: пользователь должен мочь установить десктопную версию одной командой
- **Given** система Linux/macOS/Windows
- **When** запущен `install-web.sh --desktop` (или `install-web.ps1 -Desktop`)
- **Then** бинарник `kvn-desktop` скопирован в целевой каталог, создан ярлык в приложениях
- Evidence: ярлык отображается в меню приложений/пуске, запуск открывает окно

### AC-008 Браузерный доступ сохраняется

- Почему это важно: пользователь не теряет существующий способ работы
- **Given** `kvn-desktop` запущен и показывает окно
- **When** пользователь открывает браузер на `http://localhost:2311`
- **Then** UI kvn-web доступен в браузере
- Evidence: браузер показывает полностью функциональный UI

### AC-009 Linux/macOS: кнопка Restart через polkit/osascript

- Почему это важно: пользователь может перезапустить системный сервис без терминала
- **Given** `kvn-desktop` работает, подключён к `localhost:2311`, сервис kvn-web в статусе `active`
- **When** пользователь нажимает "Restart Service" в окне kvn-desktop
- **Then** появляется системный диалог запроса пароля (polkit на Linux, osascript с administrator privileges на macOS), после ввода сервис перезапускается (`systemctl restart kvn-web` / `launchctl kickstart`)
- Evidence: после ввода пароля kvn-web перезапускается, статус в UI меняется (reconnect через SSE)

### AC-010 Windows: кнопка Restart без доп. пароля

- Почему это важно: Windows уже работает с правами администратора (UAC при запуске), повторный ввод не нужен
- **Given** `kvn-desktop` запущен с правами администратора, встроенный сервер работает
- **When** пользователь нажимает "Restart Service"
- **Then** сервер останавливается и запускается заново без запроса дополнительных прав
- Evidence: UI перезагружается (или SSE переподключается), сервис снова доступен

### AC-011 Кнопка Restart в тулбаре окна

- Почему это важно: элемент управления должен быть видимым и доступным без модификации SPA
- **Given** `kvn-desktop` запущен на любой платформе, UI загружен
- **When** пользователь смотрит на окно (его заголовок/тулбар)
- **Then** есть нативная кнопка "Restart" (через webview toolbar API или кастомный заголовок окна)
- Evidence: кнопка видна, кликабельна, при нажатии инициирует перезапуск (AC-009 / AC-010)

## Допущения

- WebView2 Runtime предустановлен на Windows 11; на Windows 10 требуется предварительная установка
- WebKitGTK (`webkit2gtk-4.1`) предустановлен или устанавливается через пакетный менеджер на Linux
- На macOS WKWebView доступен из коробки
- Порт 2311 не меняется пользователем (флаг `--port` для кастомизации)
- `kvn-web` на Linux/macOS уже настроен и работает

## Критерии успеха

- SC-001 Размер бинарника `kvn-desktop` не превышает 15 MB для каждой платформы (WebView — системный, не встраивается)
- SC-002 Время от запуска до появления окна — не более 2 секунд при уже работающем сервере
- SC-003 На Windows: полный цикл (старт сервера → окно → закрытие → cleanup) завершается без артефактов (проверка реестра, процессов)

## Краевые случаи

- Пустой state (first-run): kvn-web ни разу не запущен — экран с предложением установить сервис
- WebView2 не установлен на Windows — предложить скачать с aka.ms/webview2
- WebKitGTK не установлен на Linux — ошибка при инициализации webview
- Порт занят (Windows): `webui.Server` не может стартовать — показать ошибку
- Несколько экземпляров Windows: второй запуск может обнаружить, что порт занят, и просто открыть WebView на него (reuse)
- macOS: при запуске через Finder .app bundle может не видеть PATH для launchctl

## Открытые вопросы

- Ранее: code signing на macOS — решено: пока без подписи, ad-hoc при необходимости
- Ранее: UAC на Windows — решено: embedded manifest при сборке
- Ранее: поведение при свёрнутом окне — решено: закрытие = завершение процесса. Трей — следующий шаг
- Ранее: нейминг — решено: `kvn-desktop`
