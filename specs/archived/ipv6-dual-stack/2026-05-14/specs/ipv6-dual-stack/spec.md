# IPv6 & Dual-Stack

## Scope Snapshot

- In scope: TUN в режиме IPv6-only и dual-stack, IPv6 IP pool (fd00::/64), расширение handshake для передачи IPv6 адреса, IPv6 NAT (MASQUERADE) на сервере, раздельная маршрутизация IPv4/IPv6 пакетов.
- Out of scope: DNS64/NAT64, IPv6-only транспорт (WebSocket поверх IPv6), multi-stack балансировка, динамическое обновление правил маршрутизации для IPv6.

## Цель

Администратор VPN получает возможность развернуть туннель с IPv6-связностью (dual-stack или IPv6-only): клиенту назначается IPv6-адрес из пула fd00::/64, трафик IPv6 маршрутизируется через туннель с корректной маскарадингом на сервере. Успех измеряется работающим `ping6` и доступностью IPv6-only сайтов через туннель.

## Основной сценарий

1. Пользователь конфигурирует сервер с `pool_ipv6` (подсеть fd00::/64) и клиента с `ipv6: true` в конфиге.
2. Клиент устанавливает WebSocket-соединение, шлёт ClientHello; сервер назначает IPv4 (из pool_ipv4) и/или IPv6 (из pool_ipv6) адрес в ServerHello.
3. Клиент конфигурирует TUN-интерфейс с назначенными адресами (IPv4, IPv6 или оба).
4. Трафик IPv4 и IPv6 маршрутизируется согласно правилам (include/exclude CIDR/IP) — каждый семейство обрабатывается своим matcher'ом.
5. Сервер выполняет MASQUERADE для IPv6 трафика через nftables table `ip6`.
6. IPv6-only сайты (например, `ipv6.google.com`) доступны через туннель; `ping6` до глобальных IPv6-адресов работает.

## User Stories

- P1 Story: Как администратор, я хочу, чтобы клиенту назначался IPv6-адрес из fd00::/64, чтобы я мог развернуть dual-stack VPN.
- P2 Story: Как администратор, я хочу, чтобы IPv6-трафик клиентов проходил NAT (MASQUERADE) на сервере, чтобы клиенты могли выходить в глобальную IPv6-сеть.

## MVP Slice

Dual-stack TUN + IPv6 IP pool + расширение handshake для IPv6 + IPv6 NAT на сервере. AC-001, AC-002, AC-003, AC-004, AC-005.

## First Deployable Outcome

Сервер запускается с `pool_ipv6`, клиент подключается с `ipv6: true`, получает fd00::/64 адрес, TUN интерфейс настроен с IPv6 адресом, `ping6 fd00::<gateway>` проходит.

## Scope

- IPv6 TUN: поддержка `SetIP6` в `TunDevice`, установка IPv6 адреса через `ip -6 addr add`, MTU для IPv6.
- IPv6 IP pool: `NewIPPool6` / расширение `PoolCfg` для IPv6 подсети fd00::/64, аллокация/релиз fd00::/64 адресов.
- Расширение handshake: `AssignedIPv6` в `ServerHello`, кодирование/декодирование IPv6 адреса.
- Сессия: `AssignedIPv6` в `Session`, менеджер сессий хранит оба адреса.
- NAT: `NFTManager` настраивает также nftables table `ip6` с цепочкой postrouting masquerade для IPv6.
- Routing: `parseDstIP6` для извлечения IPv6 destination из пакета; маршрутизация по `RuleSet` для `netip.Addr` (уже поддерживает IPv6).
- Client config: `IPv6` bool флаг.
- Server config: `PoolIPv6` в `NetworkCfg`.

## Контекст

- Используется `wireguard/tun` — библиотека поддерживает TUN в режиме IPv6 и dual-stack на Linux.
- nftables поддерживает отдельные таблицы для `ip` и `ip6` family — IPv6 NAT будет в `ip6 kvn-nat`.
- `netip.Addr` в `matcher.go` уже поддерживает и IPv4, и IPv6 — маршрутизация не требует смены интерфейса Matcher.
- Сеть fd00::/64 — это Unique Local Address (ULA), маршрутизируемая только внутри туннеля (не конфликтует с глобальными адресами).
- BoltDB persistence должна учитывать IPv6 адреса наравне с IPv4.

## Требования

- RQ-001 Система ДОЛЖНА поддерживать конфигурацию IPv6 подсети (`pool_ipv6`) на сервере в формате CIDR.
- RQ-002 Система ДОЛЖНА создавать TUN-интерфейс с IPv6 адресом при `ipv6: true` на клиенте.
- RQ-003 Система ДОЛЖНА назначать клиенту уникальный IPv6 адрес из пула fd00::/64 при handshake, если IPv6 включён.
- RQ-004 ServerHello ДОЛЖЕН передавать назначенный IPv6 адрес (наряду с IPv4) в бинарном формате (16 байт family + 16 байт адрес).
- RQ-005 Сессия ДОЛЖНА хранить как IPv4, так и IPv6 назначенный адрес.
- RQ-006 NAT-менеджер ДОЛЖЕН настраивать MASQUERADE для IPv6 трафика через nftables table `ip6`.
- RQ-007 Роутер ДОЛЖЕН корректно определять IPv6 destination адрес (16 байт, начиная с offset 24) для принятия решения о маршрутизации.
- RQ-008 BoltDB persistence ДОЛЖНА корректно сохранять и восстанавливать IPv6 аллокации.

## Вне scope

- DNS64/NAT64 трансляция — не требуется, так как предполагается native IPv6.
- IPv6-only транспорт (WebSocket через IPv6) — транспорт остаётся IPv4-совместимым; IPv6-only транспорт может быть отдельной фичей.
- ICMPv6 Path MTU discovery — полагаемся на стандартное поведение ядра Linux.
- DHCPv6 / SLAAC — используется статическое назначение из пула.

## Критерии приемки

### AC-001 TUN интерфейс с IPv6 адресом

- Почему это важно: без настроенного IPv6 на TUN интерфейсе пакеты не будут маршрутизироваться.
- **Given** клиент с `ipv6: true` и сервер с `pool_ipv6`
- **When** клиент завершает handshake и получает IPv6 адрес
- **Then** на TUN интерфейсе клиента установлен IPv6 адрес (видно через `ip -6 addr show dev kvn`)
- Evidence: команда `ip -6 addr show dev kvn` показывает назначенный ULA адрес

### AC-002 IPv6 адрес назначается из пула

- Почему это важно: гарантирует уникальность и отсутствие коллизий.
- **Given** сервер с `pool_ipv6: fd00::/64` и два подключённых клиента
- **When** оба клиента проходят handshake
- **Then** каждый клиент получает уникальный IPv6 адрес из fd00::/64
- Evidence: адреса различаются и находятся в подсети fd00::/64

### AC-003 IPv6 MASQUERADE на сервере

- Почему это важно: без NAT ответные пакеты не вернутся к клиенту.
- **Given** сервер с `pool_ipv6` и клиент с `ipv6: true`
- **When** клиент посылает IPv6 пакет на внешний адрес (например, 2001:db8::1)
- **Then** пакет приходит на сервер, сервер выполняет SNAT (MASQUERADE) через nftables `ip6 kvn-nat`
- Evidence: `nft list table ip6 kvn-nat` показывает правило masquerade; tcpdump на сервере показывает src-адрес сервера

### AC-004 ping6 проходит через туннель

- Почему это важно: базовый тест IPv6 связности.
- **Given** клиент подключён с `ipv6: true`, имеет IPv6 адрес и маршрут по умолчанию через TUN
- **When** клиент выполняет `ping6 -c 1 <gateway-ipv6>` или `ping6 -c 1 <external-ipv6>`
- **Then** ping успешен (0% loss)
- Evidence: stdout `ping6` показывает `1 packets transmitted, 1 received, 0% packet loss`

### AC-005 Dual-stack routing policy

- Почему это важно: IPv4 и IPv6 трафик должны маршрутизироваться независимо согласно правилам.
- **Given** клиент с dual-stack TUN (IPv4 + IPv6), ruleset содержит include/exclude для обоих семейств
- **When** клиент отправляет пакеты на IPv4 и IPv6 адреса
- **Then** IPv4 пакеты маршрутизируются по IPv4 правилам, IPv6 — по IPv6 правилам
- Evidence: IPv4 трафик до excluded CIDR идёт напрямую, IPv6 трафик — через туннель (и наоборот)

## Допущения

- Ядро Linux поддерживает IPv6 (CONFIG_IPV6).
- nftables доступен на сервере (проверяется `nft --version`).
- `wireguard/tun` библиотека работает корректно в dual-stack режиме на целевой ОС.
- fd00::/64 ULA не конфликтует с существующей сетевой инфраструктурой.
- Размер MTU для IPv6 учитывает больший заголовок (40 байт vs 20 байт IPv4).

## Критерии успеха

- SC-001 `ping6` и IPv6-only сайты работают через туннель.
- SC-002 Dual-stack конфигурация не ломает существующую IPv4-only функциональность (регрессия).

## Краевые случаи

- IPv6 пул исчерпан — ошибка аллокации, сессия не создаётся.
- Клиент с `ipv6: false` — никаких изменений в поведении, всё работает как раньше (только IPv4).
- Сервер без `pool_ipv6` — клиент с `ipv6: true` получает только IPv4 адрес (graceful degradation).
- Клиент с `ipv6: false` на сервере с `pool_ipv6` — назначается только IPv4.
- IPv6 TUN не создаётся (нет поддержки в ядре) — ошибка логируется, fallback на IPv4-only.
