# Multi-Server Management в KVN Web UI

## Scope Snapshot

- In scope: управление несколькими конфигурациями серверов (CRUD) через веб-интерфейс с возможностью выбора активного сервера для подключения; Import/Export/QR оперируют одним выбранным сервером.
- Out of scope: управление серверами через CLI или Android; синхронизация списка серверов между устройствами; групповая операция Import/Export.

## Цель

Пользователь KVN Web UI управляет несколькими VPN-серверами (рабочий, домашний, резервный) через единый интерфейс — добавляет, редактирует, удаляет и быстро переключается между ними без ручного редактирования YAML.

## Основной сценарий

1. Пользователь открывает KVN Web UI и видит список сохранённых серверов с текущим выбранным.
2. Пользователь может выбрать другой сервер из списка — UI загружает его конфиг.
3. Пользователь может добавить новый сервер (вручную или через Import), отредактировать поля, изменить имя, удалить ненужный.
4. При нажатии Connect используется конфиг выбранного сервера.
5. Export и QR всегда работают с конфигом текущего выбранного сервера.

## User Stories

- P1 Story: CRUD серверов + переключение + Connect по выбранному серверу.
- P2 Story: импорт конфига как нового сервера; экспорт/QR выбранного сервера.

## MVP Slice

P1 Story + P2 Story (одним срезом, т.к. Import/Export/QR без multi-server сломаются).

AC-001, AC-002, AC-003, AC-004, AC-005, AC-006.

## First Deployable Outcome

Веб-интерфейс: селектор серверов в хедере, форма редактирования для выбранного сервера, кнопка удаления, Connect использует выбранный сервер. Export/QR копируют конфиг выбранного сервера.

## Scope

- Go backend: новый API-эндпоинты для списка серверов, CRUD; расширение `config.yaml` — добавление секции `servers: []`.
- React frontend: переключение между серверами, форма редактирования, импорт как новый сервер.
- Существующие эндпоинты `/api/config` (GET/POST) адаптируются под работу с выбранным сервером.
- Import (JSON textarea) создаёт новый сервер в списке.
- Export/QR — для текущего выбранного сервера.

## Контекст

- Web UI хранит конфиг в `~/.config/kvn/config.yaml`.
- В `config.yaml` добавляется секция `servers` — массив объектов, каждый с уникальным `name` (строка) и полями `ClientConfig`:
  ```yaml
  # глобальные настройки (kvn-web / client-level)
  log:
    level: info
  proxy_listen: "127.0.0.1:2310"
  # ... остальные глобальные поля ClientConfig

  # список серверов
  servers:
    - name: Home
      server: wss://home.example.com/tunnel
      auth:
        token: xxx
      transport: quic
      tls:
        verify_mode: verify
      routing:
        default_route: server
      # ... остальные поля ClientConfig для этого сервера
    - name: Work
      server: wss://work.example.com/tunnel
      auth:
        token: yyy
      transport: tcp
  ```
- При загрузке UI: глобальные поля идут в соответствующие секции, серверные поля в форму выбранного сервера.
- При подключении конфиг выбранного сервера мержится с глобальными полями (серверные приоритетнее).
- Названия серверов уникальны в пределах Web UI.

## Требования

- RQ-001 Пользователь может создать новую конфигурацию сервера с уникальным именем и полным набором полей `ClientConfig`.
- RQ-002 Пользователь может переключаться между серверами без потери несохранённых изменений (с подтверждением при наличии dirty-состояния).
- RQ-003 Импорт JSON создаёт новый сервер с дефолтным именем (и возможностью переименовать); не заменяет текущий.
- RQ-004 Экспорт и QR работают с конфигом активного (выбранного на момент операции) сервера.
- RQ-005 Имя сервера редактируется в форме как обычное поле; валидация на уникальность при сохранении.

## Вне scope

- Серверы в CLI/Android-клиенте.
- Синхронизация списка серверов между разными машинами.
- Импорт/экспорт списка серверов целиком.
- Сортировка/группировка/теги серверов.

## Критерии приемки

### AC-001 Список серверов в UI после загрузки

- Почему это важно: без списка пользователь не знает, какие серверы доступны.
- **Given** Web UI запущен и есть сохранённые серверы
- **When** Пользователь открывает страницу
- **Then** UI отображает список имен серверов, текущий выбранный подсвечен
- Evidence: в ответе `GET /api/servers` приходит массив конфигов; UI рендерит селектор

### AC-002 Переключение между серверами

- Почему это важно: ключевая user story — переключение между серверами без потери контекста.
- **Given** Пользователь находится на странице с выбранным сервером S1 (есть несохранённые изменения)
- **When** Пользователь выбирает сервер S2
- **Then** UI показывает диалог "Сохранить изменения?" с опциями Save/Discard/Cancel
- **Given** Пользователь подтверждает (Save/Discard)
- **When** Выбран сервер S2
- **Then** Поля формы загружают конфиг S2, статус подключения сбрасывается в disconnected
- Evidence: отслеживание dirty-флага в UI; форма обновляется после подтверждения

### AC-003 Создание и удаление сервера

- Почему это важно: базовый CRUD для управления серверами.
- **Given** Пользователь на странице со списком серверов
- **When** Пользователь нажимает "Add Server"
- **Then** Появляется форма нового сервера с именем по умолчанию "Server N" и полями по умолчанию
- **When** Пользователь заполняет поля и нажимает Save
- **Then** Новый сервер появляется в списке, становится выбранным, конфиг сохраняется
- **Given** Выбран сервер (не единственный)
- **When** Пользователь нажимает "Delete"
- **Then** Появляется подтверждение; после подтверждения сервер удаляется из списка, выбирается первый оставшийся
- Evidence: ответы API CRUD; UI-обновление; файл servers.yaml изменён

### AC-004 Импорт создаёт новый сервер

- Почему это важно: импорт не должен затирать существующий конфиг.
- **Given** Пользователь открыл Import (JSON textarea)
- **When** Пользователь вставляет JSON и нажимает Apply
- **Then** Создаётся новый сервер с именем "Imported <timestamp>", полями из JSON; список обновляется, новый сервер выбран
- **Given** В списке уже есть серверы
- **Then** Ни один существующий сервер не изменён
- Evidence: после импорта `GET /api/servers` возвращает исходные серверы + новый

### AC-005 Export/QR для выбранного сервера

- Почему это важно: экспорт должен работать с тем сервером, который пользователь видит.
- **Given** Выбран сервер S1
- **When** Пользователь нажимает Export
- **Then** В буфер обмена копируется JSON конфига S1 (не список)
- **When** Пользователь нажимает QR
- **Then** QR-код генерируется из JSON конфига S1
- Evidence: вызов `navigator.clipboard.writeText` или рендер QR с правильными данными

### AC-006 Connect использует выбранный сервер

- Почему это важно: Connect должен запускать именно тот сервер, который видит пользователь.
- **Given** Выбран сервер S2, форма отображает его конфиг
- **When** Пользователь нажимает Connect
- **Then** Backend сохраняет конфиг S2 в секцию `servers` в `config.yaml` (или использует напрямую оттуда), запускает подключение к S2
- **Given** После успешного подключения пользователь переключается на сервер S1
- **When** Пользователь нажимает Connect
- **Then** Текущее подключение к S2 разрывается, запускается подключение к S1
- Evidence: POST /api/connect читает конфиг выбранного сервера; при переключении сервера disconnect + connect

## Допущения

- Серверы хранятся в секции `servers: []` в `~/.config/kvn/config.yaml`.
- При первом запуске без секции `servers` создаётся один сервер "Default" с конфигом по умолчанию (существующий `config.yaml` мигрируется в `servers[0]`).
- Каждый сервер хранит полный `ClientConfig` (все поля); глобальные поля сверху — для kvn-web и client-level настроек.
- При подключении серверный конфиг мержится с глобальным (серверные поля переопределяют глобальные).
- Имя сервера (`name`) — уникальный идентификатор в пределах config.yaml.
- При удалении текущего сервера автоматически выбирается первый из оставшихся.

## Критерии успеха

- SC-001 Сервер переключается < 500ms (без reconnect).
- SC-002 При 10+ серверах UI остаётся отзывчивым, CRUD операции < 200ms.

## Краевые случаи

- Пустой список (первый запуск): создаётся сервер "Default" с конфигом по умолчанию.
- Попытка удалить последний сервер: кнопка Delete неактивна или выдаёт предупреждение.
- Импорт невалидного JSON: показать ошибку, список не изменяется.
- Импорт с дублирующимся именем: автоматически добавить суффикс " (2)".
- Переключение во время активного подключения: предложить отключиться, затем переключить.
- Редактирование сервера без сохранения и уход со страницы: предупреждение о несохранённых изменениях.
