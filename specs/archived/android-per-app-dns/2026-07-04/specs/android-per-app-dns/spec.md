# Per-App Filtering + DNS Configuration for Android VPN

## Scope Snapshot

- In scope: полноценный UI выбора приложений (include/exclude) для Android VpnService + настройка DNS-серверов на уровне приложения.
- Out of scope: split-tunnel по доменам (удалён), geoip/geosite routing sources (остаются без изменений).

## Цель

Пользователь Android-клиента получает удобный интерфейс для выбора, какие приложения направлять через VPN, а какие — напрямую, а также возможность указать кастомные DNS-серверы. Настройки сохраняются на уровне приложения (не привязаны к серверу) и не сбрасываются при переключении сервера.

## Основной сценарий

1. Пользователь открывает экран подключения.
2. В секции "Apps" видит два режима: Allowlist (только выбранные приложения идут через VPN) и Blocklist (выбранные приложения обходят VPN).
3. Нажимает "Select Apps" → открывается полноэкранный список установленных приложений с иконками, названиями, поиском.
4. Отмечает нужные приложения и подтверждает выбор.
5. В секции "DNS" указывает адреса DNS-серверов через запятую.
6. Нажимает Connect — туннель создаётся с учётом выбранных приложений и DNS.
7. Настройки сохраняются и применяются при следующем подключении, независимо от выбранного сервера.

## User Stories

- P1 Story: Пользователь может выбрать конкретные приложения для направления через VPN.
- P2 Story: Пользователь может настроить DNS-серверы для VPN-туннеля.

## MVP Slice

- Рабочий UI списка приложений с чекбоксами (allowlist OR blocklist), текстовое поле для DNS-серверов, сохранение настроек между сессиями.

## Контекст

- Настройки per-app и DNS НЕ привязаны к серверу — хранятся в AppConfig (app-level)
- `VpnService.Builder` поддерживает ТОЛЬКО один режим: `addAllowedApplication` (все выбранные через VPN) ИЛИ `addDisallowedApplication` (все выбранные НЕ через VPN). Нельзя смешивать.
- Системные приложения доступны для выбора через `PackageManager`
- Текущая реализация: текстовые поля для package name — требуется замена на App Picker

## Зависимости

- `android-per-app-dns` зависит от базовой структуры `AppConfig` (уже реализована)
- `none` — внешних зависимостей нет

## Требования

- RQ-001 Система ДОЛЖНА предоставлять UI для просмотра и выбора установленных приложений с отображением имени и иконки
- RQ-002 Система ДОЛЖНА поддерживать поиск/фильтрацию в списке приложений
- RQ-003 Пользователь ДОЛЖЕН выбрать режим: "Только эти приложения" (allowlist) ИЛИ "Кроме этих приложений" (blocklist)
- RQ-004 Система ДОЛЖНА сохранять выбор приложений на уровне приложения (не зависит от сервера)
- RQ-005 Система ДОЛЖНА принимать список DNS-серверов (IPv4, comma-separated) и передавать в `VpnService.Builder.addDnsServer()`
- RQ-006 Система НЕ ДОЛЖНА позволять одновременно использовать allowlist и blocklist — только один режим

## Вне scope

- Split-tunnel по доменам — удалён, не возвращается
- Системные приложения с особыми правами (Android ID, Google Play Services) — выбор доступен, но ответственность за корректность работы на пользователе
- Сохранение per-app настроек в QR-код — настройки device-specific, не экспортируются

## Критерии приемки

### AC-001 App Picker Screen ✅

- Почему это важно: пользователь не обязан знать package name — он выбирает приложение по имени
- **Given** пользователь на экране ConnectScreen, секция Apps
- **When** он нажимает "Select Apps"
- **Then** открывается AppPickerScreen со списком всех установленных приложений (имя + иконка + package name подпись)
- Evidence: полноэкранный список приложений отображается, скроллится, каждое приложение имеет чекбокс

### AC-002 Search/Filter ✅

- Почему это важно: при 200+ установленных приложениях поиск критичен
- **Given** AppPickerScreen открыт
- **When** пользователь вводит текст в поле поиска
- **Then** список фильтруется по имени приложения и package name
- Evidence: при вводе "chro" остаются только приложения с "chro" в имени/package

### AC-003 Mode Switch (Allowlist / Blocklist) ✅

- Почему это важно: пользователь может хотеть пустить через VPN только банковские приложения (allowlist) или исключить торренты (blocklist)
- **Given** секция Apps на ConnectScreen
- **When** пользователь переключает режим между "Allowed apps" и "Blocked apps"
- **Then** соответствующий список отображается, и в VpnService используется правильный метод Builder
- Evidence: при выборе "Allowed" вызывается `addAllowedApplication`, при "Blocked" — `addDisallowedApplication`

### AC-004 DNS Configuration ✅

- Почему это важно: некоторые провайдеры блокируют стандартные DNS
- **Given** секция DNS на ConnectScreen
- **When** пользователь вводит "77.88.8.8,77.88.8.1" и нажимает Connect
- **Then** VPN туннель использует указанные DNS-серверы
- Evidence: `VpnService.Builder.addDnsServer()` вызван с 77.88.8.8 и 77.88.8.1

### AC-005 Settings Persist Across Servers ✅

- Почему это важно: пользователь настроил apps/DNS под себя и не хочет перенастраивать при смене сервера
- **Given** пользователь настроил App Picker и DNS
- **When** он переключает активный сервер
- **Then** настройки apps и DNS остаются без изменений
- Evidence: после переключения сервера поля apps/DNS показывают те же значения

- `PackageManager.getInstalledApplications()` возвращает все приложения, включая системные

## Критерии успеха

- SC-001 App Picker открывается < 500ms на устройстве с 300+ приложениями
- SC-002 Поиск фильтрует список < 100ms после каждого нажатия клавиши

## Краевые случаи

- Пустой allowlist = весь трафик через VPN (как будто per-app не настроен)
- Пустой blocklist = весь трафик через VPN
- Все приложения в allowlist = эквивалентно "всё через VPN"
- Все приложения в blocklist = VPN активен, но ни одно приложение его не использует
- Приложение удалено после сохранения настроек → `addAllowedApplication` кинет `IllegalArgumentException`, его надо ловить
- Невалидный DNS-сервер → `InetAddress.getByName()` кинет `UnknownHostException`, сервер пропускается
