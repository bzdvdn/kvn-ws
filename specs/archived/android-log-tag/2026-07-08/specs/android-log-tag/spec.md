# Android Logging System (In-App Log Viewer)

## Scope Snapshot

- In scope: полноценная система логирования для Android-клиента: структурированные лог-записи с уровнями, in-memory буфер, и красивый in-app log viewer с живой лентой, фильтрацией, поиском, копированием, экспортом и share.
- Out of scope: чтение system Logcat, логирование на стороне Go-сервера, централизованный сбор логов (remote logging), персистентность логов между сессиями.

## Цель

Разработчик и продвинутый пользователь Android-клиента KVN получает возможность видеть логи приложения в реальном времени прямо в UI: фильтровать по уровню/тегу, искать текст, ставить ленту на паузу, копировать отдельные записи, экспортировать весь буфер в файл и делиться им. Фича считается успешной, когда старый примитивный AlertDialog с "Routing Logs" заменён на полноэкранный Log Viewer с живым потоком и все перечисленные операции работают.

## Основной сценарий

1. Пользователь переключается на 4-й таб "Logs" в Bottom Navigation (между Settings и Traffic).
2. Открывается полноэкранный Log Viewer с живой лентой логов: новые записи добавляются в реальном времени, лента auto-scroll вниз.
3. Пользователь может:
   - Фильтровать по уровню (DEBUG/INFO/WARN/ERROR) — чипсет или выпадающий список.
   - Фильтровать по тегу — список активных тегов с toggle.
   - Искать текст — SearchBar, подсветка совпадений, фильтрация.
   - Поставить паузу — лента замирает, можно скроллить историю, не отвлекаясь на новые записи.
   - Скопировать одну запись — долгий тап → копирование в clipboard.
   - Очистить буфер — кнопка Clear.
   - Экспортировать — кнопка Export → сохранение в файл (Downloads).
   - Поделиться — кнопка Share → системный share sheet с текстом логов.
4. Ошибка/fallback: при ошибке экспорта/записи файла показать Snackbar с описанием; если логов нет — пустое состояние с иконкой и подсказкой.

## MVP Slice

in-app log viewer с живой лентой + фильтр по уровню + текстовый поиск + copy одной записи + clear. Export и Share — вторая итерация.

## First Deployable Outcome

Полноэкранный Log Viewer экран (Compose) как 4-й таб Bottom Navigation, показывающий живые логи приложения с auto-scroll. Старый AlertDialog удалён. Логгер переписан на структурированные LogEntry с уровнями. Можно открыть PR и показать коллеге.

## Scope

- Создание модели данных `LogEntry` (level, tag, message, timestamp, thread).
- Создание логгера `AppLogger` в пакете `com.kvn.client.logger` с методами `d/t/w/e` и поддержкой тегов, замещающий `LogBuffer`.
- In-memory кольцевой буфер с конфигурируемой ёмкостью (default 2000).
- Live stream: `SharedFlow<LogEntry>` для подписки UI.
- Миграция всех существующих потребителей (FakeDnsResolver, KvnVpnService, QrScannerScreen) на новый логгер.
- In-app log viewer экран на Jetpack Compose + Material 3.
- Система фильтрации: по уровню (checkbox/чипсет), по тегу (multi-select), текстовый поиск.
- Pause/resume live scroll.
- Copy single entry в clipboard.
- Clear буфера.
- Export всего буфера в текстовый файл.
- Share через Android Intent.
- Пустое состояние, error state при экспорте.
- `@sk-task` / `@sk-test` маркеры traceability.

## Контекст

- Существующий `LogBuffer` (object, `com.kvn.client.dns`) — примитивный ring buffer без уровней, только строки. Живёт в неправильном пакете (dns).
- Существующий log viewer — AlertDialog в `SettingsScreen`, снэпшот без live streaming.
- Есть прямое использование `android.util.Log` в `QrScannerScreen` (3 вызова).
- Android-стек: Jetpack Compose + Material 3, Kotlin coroutines + SharedFlow.
- Требуется thread-safe доступ: логгер вызывается из разных потоков (VpnService threads, coroutines).
- Нет и не планируется remote logging — только локальный просмотр.
- minSdk = 26 (Android 8.0). Export через `getExternalFilesDir(Environment.DIRECTORY_DOWNLOADS)` (API 26-28), `MediaStore.Downloads` (API 29+); share через `FileProvider`.
- Log Viewer — отдельный 4-й таб Bottom Navigation (`NavigationBar`), расположенный между Settings и Traffic (индексы: 0=Connect, 1=Settings, 2=Logs, 3=Traffic).

## Зависимости

- `kotlinx-coroutines-core` — уже есть в проекте.
- `androidx.compose.material3` — уже есть.
- `androidx.compose.foundation` — уже есть.
- Никаких новых external библиотек (Timber, Logcat) не добавляем.

## Требования

- RQ-001 Система ДОЛЖНА предоставлять модель `LogEntry` с полями: `level` (enum: DEBUG, INFO, WARN, ERROR), `tag` (String), `message` (String), `timestamp` (Instant), `thread` (String).
- RQ-002 Логгер ДОЛЖЕН предоставлять методы `d(tag, msg)`, `i(tag, msg)`, `w(tag, msg)`, `e(tag, msg)` для записи логов.
- RQ-003 Логгер ДОЛЖЕН хранить записи в in-memory кольцевом буфере с конфигурируемым максимальным размером (default 2000).
- RQ-004 Логгер ДОЛЖЕН экспортировать `SharedFlow<LogEntry>` для подписки UI на живые логи.
- RQ-005 Логгер ДОЛЖЕН быть thread-safe (вызовы из любых потоков).
- RQ-006 Log Viewer ДОЛЖЕН показывать живую ленту логов с auto-scroll к новым записям.
- RQ-007 Log Viewer ДОЛЖЕН поддерживать паузу/возобновление auto-scroll.
- RQ-008 Log Viewer ДОЛЖЕН поддерживать фильтрацию по уровню (multi-select: показать только выбранные уровни).
- RQ-009 Log Viewer ДОЛЖЕН поддерживать фильтрацию по тегу (multi-select из списка активных тегов).
- RQ-010 Log Viewer ДОЛЖЕН поддерживать текстовый поиск с подсветкой совпадений.
- RQ-011 Log Viewer ДОЛЖЕН поддерживать копирование отдельной записи в clipboard по долгому тапу.
- RQ-012 Log Viewer ДОЛЖЕН поддерживать очистку буфера через кнопку Clear.
- RQ-013 Log Viewer ДОЛЖЕН поддерживать экспорт текущего буфера в текстовый файл (в Downloads).
- RQ-014 Log Viewer ДОЛЖЕН поддерживать share содержимого буфера через системный share sheet.
- RQ-015 При отсутствии логов Log Viewer ДОЛЖЕН показывать пустое состояние с поясняющим текстом.
- RQ-016 При ошибке экспорта Log Viewer ДОЛЖЕН показать Snackbar с сообщением об ошибке.
- RQ-017 Все существующие потребители (FakeDnsResolver, KvnVpnService, QrScannerScreen) ДОЛЖНЫ быть переведены на новый логгер.
- RQ-018 Старый `LogBuffer` ДОЛЖЕН остаться как deprecated wrapper, не удаляется (обратная совместимость).
- RQ-019 Log Viewer ДОЛЖЕН быть доступен как 4-й таб в Bottom Navigation (`NavigationBar`) между Settings и Traffic.

## Вне scope

- Чтение и отображение system Logcat.
- Remote/centralized logging (отправка логов на сервер).
- Персистентность логов между запусками приложения (сохранение в БД/файл).
- Логирование на стороне Go-сервера (Go использует zap, не трогаем).
- Web UI логи (JavaScript console).
- Шифрование/защита экспортированных логов.
- Автоматическая очистка по времени (TTL).
- Логирование network traffic (packet-level).

## Критерии приемки

### AC-001 Live log streaming

- Почему это важно: пользователь видит логи в реальном времени без ручного обновления.
- **Given** приложение запущено и производит логи (DNS resolution, TUN events)
- **When** пользователь открывает Log Viewer
- **Then** новые лог-записи появляются в списке автоматически, список auto-scroll к последней записи
- Evidence: при открытии Log Viewer во время активного VPN соединения видны новые строки каждые несколько секунд без ручного скролла

### AC-002 Filter by log level

- Почему это важно: разработчик может сузить ленту до важных сообщений.
- **Given** Log Viewer открыт, в буфере есть записи разных уровней
- **When** пользователь выбирает фильтр "только WARN + ERROR"
- **Then** отображаются только записи с уровнем WARN и ERROR, DEBUG и INFO скрыты
- Evidence: счётчик отфильтрованных записей (или количество строк на экране) меньше общего числа записей в буфере

### AC-003 Filter by tag

- Почему это важно: разработчик может изолировать логи конкретного компонента.
- **Given** Log Viewer открыт, в буфере есть записи с тегами "DNS", "TUN", "TCP", "ROUTE"
- **When** пользователь выбирает только тег "TUN"
- **Then** отображаются только записи с тегом "TUN"
- Evidence: в списке нет строк с другими тегами

### AC-004 Text search

- Почему это важно: быстрый поиск по ключевым словам в логах.
- **Given** Log Viewer открыт, в буфере есть записи
- **When** пользователь вводит поисковый запрос "timeout"
- **Then** отображаются только записи, содержащие "timeout", и текст совпадения подсвечен (визуально отличается)
- Evidence: визуальная подсветка найденного текста, количество отфильтрованных записей

### AC-005 Pause/resume auto-scroll

- Почему это важно: пользователь может читать старые логи, не отвлекаясь на новые.
- **Given** Log Viewer открыт, лента auto-scroll
- **When** пользователь нажимает Pause (или скроллит вверх)
- **Then** auto-scroll останавливается, новые записи продолжают добавляться, но список остаётся на текущей позиции
- **When** пользователь нажимает Resume (кнопка "scroll to bottom")
- **Then** auto-scroll возобновляется, список скроллится к последней записи
- Evidence: после паузы новые записи видны при ручном скролле вниз; после Resume — автоматический скролл к низу

### AC-006 Copy single entry

- Почему это важно: быстрая отправка конкретной записи в чат/тикет.
- **Given** Log Viewer открыт, в списке есть записи
- **When** пользователь долго тапает на запись
- **Then** текст записи копируется в системный clipboard, показывается Snackbar "Copied"
- Evidence: после долгого тапа вставить в другое приложение — текст записи в буфере обмена

### AC-007 Export buffer to file

- Почему это важно: пользователь может сохранить логи для анализа.
- **Given** Log Viewer открыт, в буфере есть записи
- **When** пользователь нажимает Export
- **Then** файл с содержимым буфера сохраняется в директорию Downloads, показывается Snackbar с именем файла
- Evidence: файл существует в Downloads, открывается любым текстовым редактором

### AC-008 Share logs

- Почему это важно: логами можно делиться с разработчиком через мессенджеры/email.
- **Given** Log Viewer открыт, в буфере есть записи
- **When** пользователь нажимает Share
- **Then** открывается системный share sheet с текстом логов
- Evidence: можно отправить в Telegram/Gmail — в сообщении/письме текст логов

### AC-009 Clear buffer

- Почему это важно: начать сбор логов с чистого листа.
- **Given** Log Viewer открыт, в буфере есть записи
- **When** пользователь нажимает Clear
- **Then** все записи удаляются из буфера, список показывает пустое состояние
- Evidence: после Clear экран показывает "No logs yet" (или аналогичное пустое состояние)

### AC-010 Empty state

- Почему это важно: консистентный UX при отсутствии данных.
- **Given** Log Viewer открыт, буфер пуст
- **Then** отображается пустое состояние с иконкой, заголовком и поясняющим текстом (напр. "No logs yet. Use the app to generate log entries.")
- Evidence: пустое состояние визуально отлично от сломанного/загружающегося UI

### AC-011 Export error handling

- Почему это важно: пользователь понимает причину неудачи экспорта.
- **Given** Log Viewer открыт, Storage permission отсутствует
- **When** пользователь нажимает Export
- **Then** показывается Snackbar с сообщением "Export failed: no storage permission"
- Evidence: Snackbar виден 3-5 секунд

### AC-012 Existing consumers migrated

- Почему это важно: старый код продолжает работать без дублирования.
- **Given** кодовая база до фичи (FakeDnsResolver, KvnVpnService используют LogBuffer, QrScannerScreen использует android.util.Log)
- **When** фича реализована
- **Then** все три файла используют новый AppLogger; импорт LogBuffer/android.util.Log удалён из них
- Evidence: grep по коду не показывает использование старого LogBuffer (кроме его собственного файла) и `import android.util.Log`

### AC-013 Thread safety

- Почему это важно: логгер вызывается из VpnService threads без race conditions.
- **Given** логгер инициализирован
- **When** десять потоков одновременно пишут логи (1000 записей каждый)
- **Then** все 10000 записей сохранены без потери и повреждения данных; ни один вызов не бросил исключение
- Evidence: `AppLogger.getBuffer().size == 10000` после теста (в рамках limit); содержимое каждой записи корректно

## Допущения

- Log Viewer — отдельный экран как 4-й таб Bottom Navigation (не диалог и не внутри Settings).
- Пользователь даёт Storage permission для Export (или используется MediaStore API без permission).
- `AppLogger` — синглтон (object), доступный глобально, аналогично текущему `LogBuffer`.
- Существующий буфер в 500 записей увеличивается до 2000 по умолчанию.
- Автоматическая пауза при скролле вверх (как в Logcat): пользователь скроллит вверх -> auto-scroll отключается; скролл до самого низа -> auto-scroll включается.
- Дата/время в логах: `HH:mm:ss.SSS` (как сейчас) + опционально полная дата в экспорте.
- Имя файла при экспорте: `kvn-logs-YYYY-MM-dd-HHmmss.txt`.

## Критерии успеха

- SC-001 Log Viewer открывается < 100ms (измерение Compose загрузки экрана).
- SC-002 Лента остаётся отзывчивой при 2000 записей в буфере (FPS > 55 при скролле).
- SC-003 Все существующие AC по VPN-функционалу (подключение, DNS resolution, routing) продолжают проходить — миграция логгера не ломает поведение.
- SC-004 Потребление памяти буфером < 2 MB при 2000 записей (усреднённая длина сообщения ~100 символов).

## Краевые случаи

- Пустой буфер — показать empty state.
- Очень длинное сообщение (> 10000 символов) — обрезать при отображении, сохранить полностью для экспорта.
- Не-ASCII символы (unicode, эмодзи) — корректное отображение и экспорт.
- Смена ориентации — состояние фильтров и позиция скролла сохраняются (rememberSaveable).
- Экспорт при отсутствии Storage permission — показать Snackbar с ошибкой.
- Share при пустом буфере — кнопка неактивна или показать Snackbar "No logs to share".
- Clear во время активной фильтрации — буфер очищен, фильтры сброшены или сохранены.
- Высокая частота логирования (> 100 записей/сек) — буфер не крашится, старые записи вытесняются.
