# Android Logging System — Модель данных

## Scope

- Связанные `AC-*`: `AC-001`, `AC-002`, `AC-003`, `AC-004`, `AC-006`, `AC-013`
- Связанные `DEC-*`: `DEC-001`, `DEC-002`, `DEC-003`, `DEC-004`
- Статус: `changed`

## Сущности

### DM-001 LogLevel

- Назначение: enum для severity логируемого события
- Источник истины: объявление в `LogEntry.kt`
- Инварианты: ровно 4 значения (DEBUG, INFO, WARN, ERROR), строгий порядок возрастания severity
- Связанные `AC-*`: `AC-002`
- Связанные `DEC-*`: `DEC-002`
- Поля:
  - нет полей — sealed enum
  - каждая константа имеет implicit `name: String` и `ordinal: Int`
- Жизненный цикл: static compile-time, не изменяется в runtime
- Замечания по консистентности: не применимо (enum)

### DM-002 LogEntry

- Назначение: атомарная запись лога, единица хранения и отображения
- Источник истины: `AppLogger` при вызове `d/i/w/e`
- Инварианты:
  - timestamp не null и монотонно-возрастающий (в пределах одного потока)
  - level, tag, message не null и не пусты
- Связанные `AC-*`: `AC-001`, `AC-003`, `AC-004`, `AC-006`, `AC-013`
- Связанные `DEC-*`: `DEC-002`, `DEC-004`
- Поля:
  - `level: LogLevel` — required, severity записи
  - `tag: String` — required, компонент-источник (напр. "DNS", "TUN", "TCP", "ROUTE", "QR")
  - `message: String` — required, тело сообщения
  - `timestamp: Instant` — required, момент создания записи (Instant.now())
  - `thread: String` — required, имя потока (Thread.currentThread().name)
- Жизненный цикл:
  - создаётся внутри `AppLogger.log()` при вызове публичного метода `d/i/w/e`
  - хранится в in-memory кольцевом буфере до явного clear() или вытеснения при overflow
  - удаляется при clear() или при вытеснении из буфера (LRU/FIFO: oldest first)
  - не сохраняется на диск, не переживает перезапуск приложения
- Замечания по консистентности:
  - буфер может потерять записи при overflow (это ожидаемое поведение, не баг)
  - timestamp может не быть строго монотонным при мультипоточности (разные потоки пишут в разное время)

### DM-003 LogBufferState

- Назначение: view state для LogViewerScreen, snapshot буфера с учётом фильтров
- Источник истины: вычисляется в ViewModel/компоненте LogViewerScreen при подписке на AppLogger.sharedFlow + применении фильтров
- Инварианты: отражает текущие фильтры и текущее состояние буфера
- Связанные `AC-*`: `AC-002`, `AC-003`, `AC-004`, `AC-010`
- Связанные `DEC-*`: `DEC-003`, `DEC-005`
- Поля:
  - `entries: List<LogEntry>` — отфильтрованные записи для отображения
  - `allTags: Set<String>` — уникальные теги из всего буфера (для UI фильтра тегов)
  - `isPaused: Boolean` — состояние auto-scroll
- Жизненный цикл:
  - пересоздаётся при каждом изменении буфера или фильтра (через combine(sharedFlow, filterState))
  - не сохраняется, живёт только в Compose runtime
- Замечания по консистентности:
  - кратковременная рассинхронизация между живым буфером и отображаемым состоянием невозможна (combine атомарен)

## Связи

- нет значимых межсущностных связей (LogEntry не ссылается на другие сущности; LogBufferState — агрегат для UI)

## Производные правила

- `LogEntry.displayTime`: `timestamp` форматируется в `HH:mm:ss.SSS` для отображения
- `LogEntry.highlightMatches(query)`: при поиске — возвращает SpannableString с подсветкой совпадений
- `LogBufferState.availableLevels`: всегда все 4 уровня, не вычисляется
- Если производные правила не требуются в явном виде — нет

## Переходы состояний

Только буфер:
- `write` -> любое состояние -> новое состояние (запись добавлена)
- `clear` -> любое состояние -> пусто
- `overflow` (при записи, когда буфер полон) -> удаляется самая старая запись -> новая запись добавлена
- Список переходов достаточно прост для описания выше

## Вне scope

- Персистентность на диск (SQLite, DataStore, JSON file)
- Сериализация/десериализация LogEntry (напр. для remote logging)
- Формат экспорта (.txt, .json, .csv) — чистый текст
