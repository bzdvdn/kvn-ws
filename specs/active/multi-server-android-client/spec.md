# Multi-server + Dark Theme для Android-клиента

## Scope Snapshot

- In scope: Android-клиент (kvn-android) получает возможность управлять несколькими конфигурациями серверов (создание, переключение, удаление) и тёмную тему в цветах kvn-web.
- Out of scope: синхронизация конфигураций между устройствами, облачное хранение, экспорт/импорт списка серверов (только QR одной конфигурации, как сейчас).

## Цель

Пользователь Android-клиента, который использует несколько VPN-серверов (рабочий/домашний/региональный), сможет создавать и переключать профили серверов через UI, не вводя заново все поля при каждой смене. Визуальное оформление клиента приводится к единому стилю с kvn-web (тёмная тема).

## Основной сценарий

1. Пользователь открывает приложение и видит тёмную тему с селектором серверов вверху экрана.
2. Переключает сервер из выпадающего списка — форма заполняется данными выбранного сервера.
3. Нажимает Connect — приложение сохраняет конфигурацию и подключается к выбранному серверу.
4. Может добавить новый сервер (вручную или через QR) или удалить существующий.
5. При несохранённых изменениях и попытке переключения — диалог подтверждения.

## User Stories

- P1 (Multi-server MVP): пользователь может добавить 2+ сервера, переключаться между ними и коннектиться к выбранному.
- P2 (Dark theme): пользователь видит тёмную тему в цветах kvn-web.
- P3 (QR для multi-server): сканирование QR импортирует новый сервер; экспорт показывает QR выбранного сервера.

## MVP Slice

Один сервер в списке (Default) + переключение на второй добавленный через QR. AC-001, AC-002, AC-003, AC-005 закрываются первыми.

## First Deployable Outcome

После первого implementation pass можно: запустить приложение, увидеть тёмную тему, переключить сервер через селектор, подключиться к выбранному серверу. CRUD серверов работает. Несколько серверов сохраняются между перезапусками приложения.

## Scope

- Data model: `AppConfig` с полями `activeServer: String` и `servers: List<ServerEntry>`, где `ServerEntry` содержит name + `ConnectionConfig`
- DataStore: миграция с одного JSON (текущий формат `config_json`) на структурированное хранение списка серверов
- UI: селектор серверов (выпадающий список/чип), форма редактирования полей активного сервера, кнопки Add / Duplicate / Delete
- Подтверждение: диалог при переключении с несохранёнными изменениями
- QR: импорт создаёт новый сервер с авто-именем, экспорт показывает QR активного сервера
- Тема: Material 3 dark color scheme, палитра из kvn-web
- Сортировка: по имени (A→Z), активный сервер всегда первый; если VPN не подключён — последний активный так же первый
- Traceability: `@sk-task` в изменённом Kotlin-коде, `@sk-test` в изменённых Android-тестах

## Контекст

- DataStore Preferences хранит config как единственный JSON; multi-server потребует либо перехода на несколько ключей, либо единый JSON с массивом серверов
- Текущий CRUD для одного сервера: перезапись `config_json` целиком при save; multi-server требует манипуляции с массивом
- QR-код может нести только один сервер (как сейчас); импорт добавляет новый профиль
- Android не имеет глобальной конфигурации отдельно от серверов — каждый профиль хранит полный набор настроек
- Web JSON формат QR имеет поля с null-значениями (auto_reconnect, kill_switch, reconnect, system_proxy) — парсер должен их корректно обрабатывать

## Требования

- RQ-001 Система ДОЛЖНА хранить список серверов в DataStore Preferences с активным сервером при перезапуске приложения
- RQ-002 Система ДОЛЖНА автоматически мигрировать существующий одиночный конфиг в первый сервер Default
- RQ-003 Пользователь ДОЛЖЕН иметь возможность добавить новый сервер с автоматическим именем "Server N"
- RQ-004 Пользователь ДОЛЖЕН иметь возможность дублировать существующий сервер (копия всех полей) с именем "<original> (copy)"
- RQ-005 Пользователь ДОЛЖЕН иметь возможность удалить выбранный сервер
- RQ-006 Пользователь ДОЛЖЕН иметь возможность переименовать сервер
- RQ-007 Система ДОЛЖНА при нажатии Connect сохранять и подключаться к активному серверу
- RQ-008 Система ДОЛЖНА при сканировании QR добавлять новый сервер с именем "Imported <timestamp>"
- RQ-009 Система ДОЛЖНА при экспорте QR генерировать код для активного сервера
- RQ-010 Тёмная тема ДОЛЖНА использовать Material 3 dark color scheme с цветовой палитрой kvn-web
- RQ-011 Система ДОЛЖНА корректно парсить QR-коды от kvn-web, включая поля с null-значениями (auto_reconnect, kill_switch, reconnect)
- RQ-012 Кнопки управления серверами (Add/Copy/Rename/Delete) ДОЛЖНЫ быть одного размера с иконками и семантическими цветами

## Вне scope

- Глобальные настройки (mode, MTU, log level) отдельно от per-server — каждый профиль хранит полный набор
- Drag-to-reorder сортировка — сервера сортируются автоматически по имени
- Cloud backup / синхронизация между устройствами
- Batch import/export списка серверов
- Push-уведомления о статусе

## Критерии приемки

### AC-001 Multi-server CRUD

- Почему это важно: пользователь может управлять списком серверов без потери данных
- **Given** приложение запущено в первый раз
- **When** пользователь открывает экран настроек
- **Then** отображается один сервер с именем "Default" с пустыми полями
- Evidence: селектор серверов показывает ["Default"], форма пустая

### AC-002 Добавление и переключение серверов

- Почему это важно: пользователь может переключаться между конфигурациями без потери данных
- **Given** есть минимум один сервер в списке
- **When** пользователь нажимает "Add server" и заполняет поля
- **Then** новый сервер появляется в селекторе, форма показывает его данные
- Evidence: селектор показывает два элемента, при выборе другого сервера поля формы обновляются

### AC-003 Подтверждение смены сервера

- Почему это важно: предотвращает потерю несохранённых изменений
- **Given** текущий сервер имеет несохранённые изменения (dirty=true)
- **When** пользователь выбирает другой сервер из селектора
- **Then** появляется диалог "Save & Switch / Discard & Switch / Cancel"
- Evidence: диалог отображается, после выбора действие выполняется

### AC-004 Connect использует активный сервер

- Почему это важно: кнопка Connect должна коннектиться к выбранному серверу
- **Given** выбраны сервер "Work" с адресом wss://work.example.com и сервер "Home" с адресом wss://home.example.com
- **When** пользователь выбирает "Home" и нажимает Connect
- **Then** приложение подключается к wss://home.example.com
- Evidence: в логах виден адрес wss://home.example.com, трафик идёт через home-сервер

### AC-005 Тёмная тема kvn-web

- Почему это важно: единый визуальный стиль с веб-версией
- **Given** приложение запущено
- **Then** все экраны используют тёмную палитру: фон #161616, поверхности #222, текст #d0d0d0, акцент #1a5a9e, успех #2e7d32, ошибка #b71c1c
- Evidence: скриншоты и визуальный осмотр подтверждают соответствие палитре kvn-web

### AC-006 QR импорт добавляет новый сервер

- Почему это важно: QR-конфиг с другого устройства не перезаписывает существующие сервера
- **Given** есть сервер "Default"
- **When** пользователь сканирует QR-код с конфигурацией другого сервера
- **Then** создаётся новый сервер "Imported <timestamp>" с данными из QR, "Default" остаётся без изменений
- Evidence: в селекторе два сервера, старый и новый

### AC-007 QR экспорт активного сервера

- Почему это важно: экспорт должен работать для выбранного, а не единственного сервера
- **Given** у пользователя есть сервера "A" и "B", активен "B"
- **When** пользователь нажимает Export QR
- **Then** QR-код содержит конфигурацию сервера "B"
- Evidence: сканирование полученного QR восстанавливает конфиг сервера "B"

### AC-008 Duplicate server

- Почему это важно: копирование конфига для создания варианта с небольшими правками
- **Given** есть сервер "Work" с заполненными полями
- **When** пользователь нажимает "Duplicate"
- **Then** создаётся копия "Work (copy)" с теми же полями, автоматически переключается на неё
- Evidence: селектор показывает "Work (copy)", форма содержит те же данные

## Допущения

- DataStore Preferences остаётся хранилищем (не мигрируем на Room/SQLite)
- Миграция для существующих пользователей: текущий `config_json` становится сервером "Default"
- Каждый профиль сервера хранит полный набор полей ConnectionConfig (нет разделения на global/per-server)
- Тёмная тема не переключается на светлую — всегда тёмная (как kvn-web)
- Material 3 `darkColorScheme` + кастомные цвета достаточны для реализации

## Критерии успеха

- SC-001 При переключении между 3 серверами с полными конфигами задержка перерисовки формы < 200ms
- SC-002 После миграции существующий пользователь не теряет свои настройки

## Краевые случаи

- Первый запуск: список пуст → создаётся "Default" с стандартными значениями
- Удаление последнего сервера: запрещено (кнопка Delete disabled при единственном сервере)
- Удаление активного сервера: автоматическое переключение на первый в списке
- Duplicate последнего/единственного сервера: разрешено (копия "Default (copy)")
- Duplicate при dirty=true: сначала Save или Discard (как при переключении), затем Duplicate
- QR импорт с дублирующимся конфигом: всё равно создаётся новый сервер (пользователь удалит руками)
- Переименование в пустое имя: не допускается (name обязателен)
- Сортировка: по алфавиту (A→Z), активный сервер всегда первый; если VPN не подключён — последний активный первый
- Web JSON QR с null-полями (auto_reconnect и др.): поля объявлены как `Boolean?` с fallback на `true`
